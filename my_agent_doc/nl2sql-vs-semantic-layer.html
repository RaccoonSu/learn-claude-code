NL2SQL vs 语义层 API

当 AI Agent / 业务用户需要访问企业数据时，有两种截然不同的技术路径。 一种让 LLM 直接生成 SQL，另一种在数据之上构建统一的语义模型层。 本图从架构原理、优缺点和适用场景出发，给出可操作的选型参考。

第 1 层 · 架构原理

两种架构，两种信任模型

核心差异：语义层在 LLM 和数据仓库之间插入了统一、可治理的中间层，NL2SQL 则让 LLM 直接面对原始表结构。

⚡ NL2SQL 架构 直接路径
用户
自然语言查询　"上月 GMV 是多少？"
LLM 生成 SQL（黑盒）＋ 表结构 / 上下文
SELECT SUM(amount) FROM orders WHERE created_at BETWEEN ...
直接执行 SQL
无中间核对 · LLM 理解偏了，结果就不符你的本意
数据仓库
MySQL / StarRocks
✕ 意图容易跑偏：自然语言有歧义，LLM 可能会错意，给出"看起来对、其实答非所问"的结果
✕ SQL 一次性生成，无语义校验，结果不可复现
✕ 指标定义分散在 prompt / BI 工具中，无单一事实来源
✕ LLM 可能绕过权限意图（SQL 注入、越权查表）
🛡️ 语义层 API 架构 治理路径
用户
自然语言查询　"上月 GMV 是多少？"
LLM 理解语义 → 结构化参数
把问题拆成：用什么指标、按什么维度、怎么过滤　只给参数，不写 SQL
语义层（统一指标 + 关系 + 权限）
按预定义的指标把参数翻译成正确查询，同一个问题永远同一个答案
指标怎么算、表怎么连、谁能看什么 —— 都在这里定义一次
生成受控查询
按你的身份自动过滤 —— 能看什么数据，由权限说了算
数据仓库
MySQL / StarRocks
✓ 指标与 Join 关系预先定义，同一问题始终返回一致结果
✓ AI Agent 通过语义层 API 结构化发现实体，而非猜测表结构
✓ 安全策略在语义层执行，权限绕过的攻击面大幅缩小

第 2 层 · 优缺点量化

优点与缺点，逐项对照

以下对比基于架构原理和技术特征推导，不依赖单一厂商的自报基准数字。

✅ NL2SQL 的 4 个优点
🚀
启动成本极低
无需预定义指标模型或 YAML 配置，接入数据库即可开始提问。
🧠
查询灵活度高
用户可以问任何问题，不受预定义维度和度量的限制。
📦
无额外中间件
架构简单，不引入新的服务依赖和运维成本。
🔧
迭代速度快
Prompt 调优或模型升级即可改善效果，无需修改数据基础设施。
❌ NL2SQL 的 4 个缺点
🎲
结果不可复现
同一问题每次生成的 SQL 可能不同，审计困难。
📊
无单一事实来源
指标定义分散在各个 prompt、BI 工具和对话历史中。
🔓
安全边界模糊
LLM 可被诱导生成危险 SQL（删表、越权查询、注入）。
💸
幻觉放大数据错误
当 LLM 猜错表名/列名/Join 关系时，产生的是看起来合理的错误结果。
✅ 语义层 API 的 4 个优点
🎯
结果确定性
同一业务问题始终映射到同一指标定义，查询结果可复现、可审计。
🏛️
统一指标治理
Metrics/Semantic Models 是唯一事实来源，所有下游工具消费同一逻辑。
🛡️
结构性安全
行级安全、列可见性在语义层定义并强制生效，不受 LLM 行为波动影响。
🤖
AI Agent 原生兼容
Meta API / MCP Server 提供结构化实体发现能力（已验证的差异化优势）。
❌ 语义层 API 的 4 个缺点
🏗️
前期建模投入大
需要数据团队手动定义 Entities、Measures、Dimensions、Joins。
🔒
灵活性受限
用户只能查询语义模型中已定义的指标，超出范围的问题无法回答。
⚙️
运维复杂度
语义层是新的基础设施组件，需要部署、监控、版本管理和升级。
⚠️
并非零故障
历史漏洞（如 CVE-2022-23510，CVSS 9.6）证明语义层的安全边界也可能被绕过。
第 3 层 · 关键维度深挖

6 个维度，逐层判析

每个维度不是一个简单的"谁赢"，而是技术取舍。以下判断基于架构原理，而非厂商营销数据。

🎯
语义层占优

准确性与幻觉

语义层通过确定性映射消除 LLM "猜测"表结构和 Join 关系的环节。NL2SQL 的核心风险在于：LLM 生成的 SQL 可能语法正确但语义错误。语义层的风险和瓶颈转移到"指标定义本身是否准确"上。

📐
语义层占优

指标一致性

语义层的核心价值主张：一次定义，处处一致。但需注意——这一优势的前提是团队严格执行"指标只在语义层定义"的纪律，且复杂指标（半可加、多币种）仍需要仔细设计。

🔐
语义层占优

安全与治理

语义层的结构性安全优于 NL2SQL 的提示性安全。但在生产环境中，语义层也需要持续的 CVE 监控和补丁管理。历史漏洞提醒我们：没有完美安全。

💵
各有千秋

性能与成本

NL2SQL 的 API 成本主要在 LLM 调用；语义层需要额外的中间件运维成本。语义层的预聚合/缓存可以降低查询成本，但这与语义层本身是独立的能力，NL2SQL 也可以搭配缓存。

🔍
各有千秋

可解释性

语义层的指标溯源路径清晰（指标定义 → SQL 模板 → 查询结果）。NL2SQL 需要"反向解释"——从生成的 SQL 倒推 LLM 为什么这样生成，这在复杂场景下很困难。

⏱️
NL2SQL 占优

实施周期

NL2SQL 可以在几小时内搭建原型。语义层需要数周到数月的指标梳理和语义建模，适合已有成熟数据治理体系的组织。

第 4 层 · 怎么选

选型决策，看场景说话

没有绝对正确的架构，只有适合你当前阶段的架构。

场景推荐理由
快速 POC / 内部探索 NL2SQL 启动成本最低，几小时即可验证价值。适合"先跑通再治理"的阶段。
AI Agent 对接企业数据 语义层 Meta API / MCP Server 的结构化自省是目前唯一经多来源验证的差异化能力。NL2SQL Agent 在复杂 schema 上表现不稳定。
多 BI 工具指标统一 语义层 这是语义层的核心价值场景。前提是组织能投入建模和维护。
高安全/合规要求 语义层 结构性安全优于提示性安全。但不等于零风险，需要持续的安全运维。
高频 Ad-hoc 探索式查询 NL2SQL 语义层的预定义模型可能不够灵活，NL2SQL 更适合"没有标准答案"的探索。
成熟组织长期建设 组合使用 语义层作为可信基座（95% 的标准化查询），NL2SQL 作为弹性补充（5% 的探索式查询）。两者不是互斥的。
💡 最重要的建议：不要相信任何厂商的单方面基准数字。 在你的数据和场景上做 A/B 测试。语义层 + NL2SQL 不是二选一，而是可信数据基座的两个互补组件 —— 语义层提供治理锚点和AI Agent 可发现性， NL2SQL 提供灵活性和快速启动。
