01 · Agent LOOP 架构

传统 AI 助手会回答，
Agent 会动手。

我们先思考一个问题，传统 AI 助手（比如豆包）和 Agent（比如 AI 风控官、龙虾、还有开发常用的编程 agent） 到底有什么区别？——传统 AI 助手它会回答用户的问题，但不会干活，而Agent 它会动手干活。这是最核心的区别。下面我们看一个具体的例子。

第 1 步 · 先看清区别

同样一个问题，差别在哪？

用户问："把这个月的销售订单汇总，找出销售额前三的产品，做成一份简报。"

☹ 传统 AI 助手

🙋 我把这个月的销售订单汇总，找出前三，做成简报。
💬 传统 AI 助手你可以这样：① 打开订单表；② 用透视表按产品汇总销售额；③ 排序后取前三。要不要我一步步教你？
🛑 对话到此结束 · 它给完建议就停了，在等用户去操作

它会告诉用户"可以这样做"，但不会替用户做。表格得用户自己开，数据得用户自己算。

☺ AI Agent

🙋 我把这个月的销售订单汇总，找出前三，做成简报。
🤖 Agent打开订单表 → 读到 823 条订单
🤖 Agent按产品汇总并排序 → 算出前三
↑ 自己看结果、继续往下做 ↓
🤖 Agent · 已生成本月销售简报
① 产品A ¥186 万占 32%
② 产品B ¥142 万占 24%
③ 产品C ¥98 万占 17%
📊 分析：前三合计占总额 73%，产品A 持续领跑，建议保库存、追加推广。

用户只交代一次，它自己查资料、跑流程、看结果、再继续，直到把事办完。

💡 传统 AI 助手给完建议就走；Agent 是用户交代一次任务，它自己干到底、再交付。
接下来我们看一个具体的案例：传统 AI 助手给完建议就走，Agent 是说一次它自己干到底。你看左边的传统AI，它就说你可以打开订单表、用透视表汇总、排序取前三，给完建议就停了，等你操作。右边的 Agent 呢？它自己打开订单表、读到八百多条订单，自己汇总排序，最后直接把简报交给你——前三各占多少、总占比、还给分析建议。全程你就说了一句话。所以 Agent 和 Chatbot 的本质区别就在于——一个给建议，一个出结果。那 Agent 是怎么做到自己干到底的？翻下一页看原理。

第 3 步 · 看真实例子

它在真实任务里是怎么"自己跑"的？

用户交代："把本月销售订单汇总，找出销售额前三，做成简报。"
① Agent 理解用户问题，决定先打开订单表。
"我要用工具" → 打开本月销售订单表
② 看到数据：共 800 多条订单。
→ 自己决定下一步：按产品汇总并排序
③ Agent 再次动手，算出销售额、排出前三。
"我要用工具" → 执行：计算并排序
④ 看到结果，写成带结论的简报交给用户。
"我做完了" → 输出简报，循环结束

对我们的业务，意味着什么？

🔄 从"一问一答"到"一问到底" —— 交代一次目标，它推进到完成
🧩 能自己拆解·执行·纠错 —— 遇到报错自己看、自己改、继续走
🚀 替代重复性劳动 —— 运维、报表、核查从"人做"变"Agent做"
👀 全程用户只说了一句话。中间的打开表格、看数据、算排名、再决定，全是 Agent 自己完成的 —— 这就是Agent loop在起作用。
还是那个销售订单任务。Agent 第一步想了一下先打开订单表。第二步看到八百多条数据，自己决定要汇总排序。第三步再次动手计算排名。第四步看到结果写成简报交给你。四步下来全程一句话没多说。对业务来说：从"一问一答"升级到"一问到底"——Agent 能拆解、执行、纠错，不需要人传话。报表、核查、运维，从人做变 Agent 做。好，接下来看主子 Agent 架构。

02 · 主子 Agent 架构

大事拆小，
分域专家会诊。

AI 怎么像一个专家委员会，让市场研判、经营分析、客商履约
各路专家分头研判、汇总结论。

好，第二个模块——主子 Agent 架构。核心思想很简单：大事拆小，像专家委员会一样分域会诊。

AI 风控官 · 主子 Agent 架构

第 1 步 · 先看为什么要拆

一个 AI 啥都管，为什么看不深、还健忘？

让一个 AI 同时懂市场、懂经营、又懂履约，它的脑袋会被各种中间过程塞满——哪个领域都看不深，还会走神、忘记最初要研判什么。就像一个人想同时当三个专家，结果哪个都不精。

☹ 一个 AI 全包

市场、经营、履约全塞进同一个脑袋：读的数据、算的过程、试错的来回……越塞越满，每个域都只能浅尝辄止，还忘了目标。

记忆被占满 → 看不深、健忘、跑偏。

☺ 拆成分域专家

每个领域交给一位专家，只盯自己那块、用自己的专业口径深挖。专家做完汇报研判结论，主管综合各方做判断。

各域深挖 + 主管汇总 → 看得深、看得全。
为什么要把事拆开？你想，让一个 AI 同时懂市场、懂经营、又懂履约，脑袋里要装的东西太多了——读的数据、算的过程、各种试错来回，塞得满满的。结果每个领域都只能浅尝辄止，还容易走神、忘了最初要研判什么。怎么办？拆！市场交给市场专家，经营交给经营专家，履约交给履约专家。每位专家只盯自己那块，用自己的专业口径深挖，做完只汇报结论。主管拿各方结论做综合判断。翻下一页看架构图。

第 3 步 · 为什么这样设计 + 实战

两个关键设计，让它真的像个专家委员会

🧠 子Agent 独立上下文空间

每位子Agent 有独立的上下文空间和专业口径，互不干扰。市场专家和履约专家各自研判，数据不共享。

独立上下文空间 互不干扰 独立研判

📋 主Agent 与子Agent 上下文隔离

子Agent 研判完只把结论交回统筹，中间数据全部清除。主Agent 上下文不受子Agent 过程污染。

主子之间上下文隔离 仅保留关键结论

案例："研判本季度经营状况"

🙋 用户研判本季度整体经营状况，给出综合判断和风险提示。
🧑‍💼 统筹（主 Agent）收到。请市场研判专家分析外部市场与竞争态势，请经营分析专家分析内部经营数据，请客商履约专家分析客户履约情况和风险。
📨 市场研判专家行业需求收缩，价格战加剧 → 外部承压
📨 经营分析专家营收增长但毛利下滑，成本上升 → 增收不增利
📨 客商履约专家履约整体正常，某大客户回款逾期 → 局部回款风险
🧑‍💼 统筹综合三方：「外部承压下的增收不增利」，建议控成本、盯回款、守份额。
两个关键设计。第一，子Agent 独立上下文空间——每位专家有独立的上下文和专业口径，互不干扰，独立研判。第二，主Agent 与子Agent 上下文隔离——子Agent 只回结论，仅保留关键结论，中间数据全部清除，主 agent上下文不受污染。看下面案例：用户说研判本季度经营状况，统筹派给三位专家。市场研判专家回报外部承压，经营分析专家回报增收不增利，客商履约专家回报局部回款风险。统筹综合三方得出最终判断——外部承压下的增收不增利，建议控成本、盯回款、守份额。好，接下来讲Skill 技能。

第 1 步 · 先搞懂一件事

大模型没有记忆，每次都得重新教

大模型本身没有任何记忆，每次请求对它都是全新的开始，上次聊过什么、用户教过它怎么做，它一概不记得。所以想让它按某套口径办事，每次都得把"怎么做"重新告诉它一遍。

☹ 全量加载：每次都把所有 Skill 塞进上下文

既然每次都要带，那就全带？用户只查经营指标，却把风险监控、库存分析、签约数据全部加载进上下文——响应慢，Token 消耗大，还容易幻觉。

全部 Skill 随请求发送 → 又慢又贵又容易幻觉

☺ 渐进式加载：先给目录，用到才激活

请求只带一份轻量目录索引。专家判断"这次分析经营指标"，只激活《财务重点经营指标分析》——只加载1个Skill，其余不进上下文。

用到才加载 → 省 Token、不占上下文、更不容易幻觉

Skill 就是把"怎么做"沉淀为可复用的指令模块，按需加载到上下文。以经营分析专家为例，它关联了 4 个 Skill：《财务重点经营指标分析》《风险重点监控指标分析》《库存分析》《签约数据分析》。

先搞懂一件事：大模型本身没有任何记忆，每次请求对它都是全新的开始，上次聊过什么、教过它怎么做，它一概不记得。所以想让它按某套口径办事，每次都得把"怎么做"重新告诉它。Skill 就是把规程沉淀为可复用的指令模块。但不能每次都把所有 Skill 全加载——左边笨办法，只查经营指标却把风险监控、库存、签约全带进上下文，响应慢又容易幻觉。右边渐进式加载：先只给目录索引，很轻；专家判断是经营指标分析，只激活《财务重点经营指标分析》，其余 Skill 不进上下文。用到才加载，用不到的绝不带——省成本、不占上下文，也更不容易幻觉。翻下一页看架构图。

第 3 步 · 实战 + 业务价值

经营分析专家怎么用 Skill？

用户交代："分析一下本月重点经营指标。"

对话过程

📊 这次分析经营指标，加载《财务重点经营指标分析》技能。
📖 技能加载 → 要点：①收入/毛利/费用率一起看 ②同比+环比对照 ③圈出异动指标。
✅ 按技能分析完：收入同比 +X%，但毛利降、费用率升。结论：增收不增利。
👀 专家只加载当下用得上的那一个 Skill，其余不进上下文。按需加载——既省成本，也不容易被无关内容干扰。

📚 规范自动遵守

每个领域的方法论拆成独立 Skill，专家按需加载，不用人每次提醒，质量更稳。

💰 按需加载，又省又准

不预加载无关 Skill，用到才激活。少了无关内容干扰——既更省，也更不容易跑偏胡编。

🔧 加 Skill 就能扩展

来个新规范？加一个 Skill 就行，不用动 AI 本身。Skill 越积越多，AI 越用越在行。

真实场景：用户说分析本月重点经营指标。经营分析专家判断这次是经营指标分析，激活《财务重点经营指标分析》Skill。Skill 包含三个要点：收入毛利费用率一起看、同比环比对照、圈出异动。分析完结论增收不增利。全程只激活一个 Skill。业务价值：规范自动遵守——方法论文档化为 Skill 指令，按需激活自动执行；按需加载又省又准——不预加载无关 Skill 少干扰；加 Skill 就扩展——新规范加一个 Skill 就行，不动 Agent 本身。好，接下来看语义模型。

04 · 语义模型 vs 传统文生SQL

两种架构，
两种信任模型

当 AI Agent / 业务用户需要访问企业数据时，有两种截然不同的技术路径：
一种让 LLM 直接生成 SQL 查数据库，另一种在数据之上构建统一的语义模型层。

第四个模块，一个更底层的话题——语义模型和传统文生SQL的对比。当 AI Agent 需要访问企业数据时，有两条截然不同的路。一条让 LLM 直接写 SQL，另一条在数据之上构建统一的语义模型层。这是两种完全不同的信任模型，接下来我们来看下架构原理。

第 1 层 · 架构原理

两种架构，两种信任模型

传统文生SQL 架构 · 直接路径

用户
"本月销售额前三的产品是哪些？"
LLM 生成 SQL（黑盒）＋表结构/上下文
SELECT product, SUM(amount) FROM orders GROUP BY product ORDER BY 2 DESC LIMIT 3
直接执行 SQL
无中间核对 · LLM 理解偏了，结果就不符本意
数据仓库
MySQL / StarRocks
核心问题：LLM 生成的 SQL 语法正确但语义偏离预期，表 Join 关系容易混乱，结果"看起来对，实际答非所问"
自然语言歧义 + 无语义校验，同一问题每次生成 SQL 可能不同，结果不可复现
指标定义分散在 prompt / BI 工具中，无单一事实来源
LLM 可能绕过权限意图（SQL 注入、越权查表）

语义模型架构 · 治理路径

用户
"本月销售额前三的产品是哪些？"
LLM 理解语义 → 结构化参数
拆成：用什么指标、按什么维度、怎么过滤　只给参数，不写 SQL
语义层（统一指标 + 关系 + 权限）
按预定义指标把参数翻译成正确查询，同一问题永远同一答案
指标怎么算、表怎么连、谁能看什么 —— 都在这里定义
生成受控查询
按身份自动过滤 —— 能看什么数据，由权限说了算
数据仓库
MySQL / StarRocks
指标与 Join 关系预先定义，同一问题始终返回一致结果
AI Agent 通过语义模型结构化发现实体，而非猜测表结构
安全策略在语义层执行，权限绕过的攻击面大幅缩小
用户 LLM 组件 风险步骤 语义层组件 数据仓库
左边传统文生SQL，用户问"本月销售额前三的产品是哪些"，LLM 直接生成 SQL 去查数据库，中间没有核对。容易出现什么问题？意图跑偏、SQL 语法对但语义错、指标定义分散、还可能 SQL 注入越权。右边语义模型架构，多了一层——LLM 只负责把自然语言转成结构化参数，只给参数不写 SQL；语义层拿到参数，结合预定义的指标、表关系和权限，翻译成正确查询。同一个问题永远返回一样答案，权限也在语义层执行。核心差异就是直接路径和治理路径。好，接下来看图表工具。

第 3 步 · 两个亮点

它比"画个图"多做了什么

🧠 自动选图

选图小专家（LLM）只看抽样几行+意图，输出出图规则：用什么图、X/Y怎么放。"说人话就行"，不用人指定。

🗃️ 不撑爆上下文

LLM 上下文空间有限，不适合一次传入全部数据（比如 100 行）。我们的做法：选图阶段只传抽样 5 行给 LLM 判断、分析，并给出数据解析规则，全量数据由程序按规则处理，不进模型上下文——既不撑爆窗口，也不会被截断。

两个亮点：第一自动选图——选图小专家只看抽样加意图就输出出图规则，说人话就行；第二不撑爆上下文——LLM只喂抽样，全量数据由程序处理，不进模型上下文。好，最后一个模块——卡片工具。
