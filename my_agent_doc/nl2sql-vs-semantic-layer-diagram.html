NL2SQL vs 语义层 API

让 AI Agent / 业务用户访问企业数据的两条路径 · 架构与可信度对比
⚡ 直接路径 · 灵活但不可控 🛡 治理路径 · 受控但需建模
⚡

NL2SQL 架构

直接路径 · LLM 即信任锚点
用户 / AI Agent
自然语言查询　"上月 GMV 是多少？"
▼
LLM 生成 SQL（黑盒）
SELECT SUM(amount) FROM orders
WHERE created_at BETWEEN ...
▼
直接执行 SQL
无中间校验 · 裸查询
▼
数据仓库
BigQuery / Snowflake / …
✕
SQL 一次性生成，无语义校验，结果不可复现
✕
指标定义分散在 prompt / BI 工具中，无单一事实来源
✕
LLM 可能绕过权限意图（SQL 注入、越权查表）
!
失败静默且可信——错误数字可能流入汇报无人察觉
VS
🛡

语义层 API 架构

治理路径 · 语义模型即信任锚点
用户 / AI Agent
自然语言查询　"上月 GMV 是多少？"
▼
LLM 理解人话 → 结构化参数
指标 · 维度 · 限定 · 排序 · limit
输出参数对象，不输出 SQL
▼
语义层 Meta API
结构化自省 · discover_entities
execute_query 路由至预定义指标 → 锚定性解析
统一语义模型（人工治理，一次定义）
指标定义 · Join 关系 · 权限策略
▼
生成受控查询
经过校验 · 权限已应用
▼
数据仓库
BigQuery / Snowflake / …
✓
指标与 Join 关系预定义，同一问题始终一致
✓
Agent 通过 Meta API 结构化发现实体，而非猜测表结构
✓
安全策略在语义层执行，权限绕过攻击面大幅缩小
✓
失败嘈杂——越界 / 不可解析会报错并阻断传播
🎯
核心权衡：灵活性 ↔ 可信度。NL2SQL 实施快、灵活，适合 ad hoc / 小数据集 / 有人类复核；语义层准确、一致、安全、可治理，适合企业级 / 对外汇报 / 监管口径。 两者互补而非互斥——生产建议：语义层做治理边界 + 仓库原生 RLS 兜底 + NL2SQL 处理范围外长尾（配验证 agent）。
